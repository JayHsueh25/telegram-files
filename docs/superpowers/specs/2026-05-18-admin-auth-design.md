# 管理员账号密码登录设计

## 1. 目标

为 `telegram-files` 增加一套最小可用的站点级管理员登录能力，满足以下目标：

- 未登录时不能使用站内功能
- 系统仅支持一个管理员账号
- 首次启动提供初始账号密码：`admin / admin`
- 登录后可在页面内修改账号和密码
- 修改账号密码时不要求再次输入旧密码
- 认证能力同时约束页面访问、HTTP 接口访问和 WebSocket 连接

本次只实现站点管理员认证，不引入多用户体系、角色体系或第三方认证。

## 2. 背景与现状

当前项目前端位于 `web`，基于 Next.js App Router；后端位于 `api`，基于 Vert.x。

现状中已经具备以下基础能力：

- 后端已经启用了 `SessionHandler`，具备 Cookie Session 基础设施
- 前端请求工具默认携带 `credentials: "include"`，可以直接复用 Cookie Session
- 后端已经有通用设置表 `setting_record`，适合存放管理员账号和密码哈希
- 前端已有统一设置弹窗，可承载“账号安全”配置页签

因此，本次最合适的方案是直接基于现有 Session 和设置表补一层站点管理员认证，而不是另起一套用户系统。

## 3. 方案对比

| 方案 | 描述 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| A | 后端 Session + 设置表存管理员凭据 + 前端登录页与拦截 | 真正保护页面、接口和 WebSocket；复用现有能力；无需新增依赖 | 前后端都要改 | 采用 |
| B | 仅前端本地登录页，本地存储账号状态 | 开发快 | 不能保护 API 和 WebSocket，安全性不足 | 不采用 |
| C | 完整多用户认证系统 | 扩展性更强 | 超出当前需求，范围过大 | 不采用 |

## 4. 范围

### 4.1 本次包含

- 新增管理员登录页
- 新增登录状态查询、登录、退出登录、修改账号密码接口
- 增加后端认证拦截器，保护业务接口和 WebSocket
- 前端增加登录态守卫
- 在设置弹窗中新增“账号安全”页签
- 首次初始化自动写入初始账号密码

### 4.2 本次不包含

- 多账号管理
- 权限分级
- 忘记密码找回
- 二次验证
- 登录失败次数限制
- 首次登录强制改密

## 5. 认证模型

### 5.1 账号模型

- 系统只有一个管理员账号
- 管理员账号和密码哈希保存在设置表中
- 账号名默认是 `admin`
- 初始密码默认是 `admin`

### 5.2 会话模型

- 登录成功后，在当前 Session 中写入管理员已认证标记
- 后续 HTTP 请求依赖 Session Cookie 识别登录态
- WebSocket 握手时也校验同一个 Session
- 退出登录时清除 Session 中的认证标记

### 5.3 凭据更新模型

- 修改账号和密码时，要求当前会话已登录
- 不要求再次输入旧密码，这是基于本次确认结果确定的产品行为
- 修改成功后，当前会话保持登录，不强制重新登录

## 6. 凭据存储设计

### 6.1 设置项

在 `SettingKey` 中新增以下键：

- `adminUsername`
- `adminPasswordHash`

如果需要单字段封装哈希元数据，也可以将盐值和参数一并编码进 `adminPasswordHash`，避免额外增加设置项。

### 6.2 密码保存方式

密码不以明文存储。后端使用 JDK 自带的 `PBKDF2WithHmacSHA256` 生成密码哈希，并将以下信息编码保存：

- 算法标识
- 迭代次数
- 随机盐
- 哈希结果

推荐持久化格式：

```text
pbkdf2$<iterations>$<base64-salt>$<base64-hash>
```

这样后续校验和升级都比较直接，也不需要新增依赖。

### 6.3 初始值写入策略

系统启动后，如果设置表里不存在管理员账号或密码哈希，则自动初始化：

- `adminUsername = admin`
- `adminPasswordHash = admin` 对应的 PBKDF2 哈希

初始化逻辑只在“当前没有管理员凭据”时执行一次。后续即使服务重启，也不覆盖用户手动修改后的值。

### 6.4 环境变量

为兼顾部署灵活性，支持以下环境变量作为“首次初始化默认值”的覆盖项：

- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`

默认值仍然是 `admin / admin`。只有在数据库里尚未存在管理员凭据时，这两个环境变量才生效。

## 7. 后端设计

### 7.1 接口设计

新增认证相关接口：

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `POST` | `/auth/login` | 管理员登录 |
| `POST` | `/auth/logout` | 退出登录 |
| `GET` | `/auth/session` | 获取当前登录状态 |
| `POST` | `/auth/credentials` | 修改管理员账号和密码 |

#### `POST /auth/login`

请求体：

```json
{
  "username": "admin",
  "password": "admin"
}
```

返回：

- 成功：`200`
- 失败：`401`，统一返回“账号或密码错误”

行为要求：

- 校验账号和密码是否匹配
- 匹配成功后将登录标记写入 Session
- 不泄漏“账号错了”还是“密码错了”

#### `GET /auth/session`

返回示例：

```json
{
  "authenticated": true,
  "username": "admin"
}
```

用途：

- 前端初始化时判断是否已登录
- 页面跳转守卫和登录页重定向都依赖这个接口

#### `POST /auth/logout`

行为：

- 清除当前 Session 登录标记
- 返回 `200`

#### `POST /auth/credentials`

请求体：

```json
{
  "username": "new-admin",
  "password": "new-password"
}
```

行为：

- 要求当前已登录
- 校验新账号非空
- 校验新密码非空，并满足最小长度约束
- 更新设置表中的管理员账号和密码哈希
- 成功后保持当前会话登录
- 如当前 Session 中缓存了用户名，也同步更新

### 7.2 认证拦截

在 `HttpVerticle` 中新增统一认证守卫，思路如下：

- 允许匿名访问的路径：
  - `/`
  - `/health`
  - `/version`
  - `/auth/login`
  - `/auth/session`
- 受保护路径：
  - 所有其他业务接口
  - `/ws`

说明：

- `/auth/session` 对未登录场景也要可访问，否则前端无法判断当前状态
- 根路径 `/` 目前只是 API 的 Hello World，不承载业务能力，可保持匿名访问
- 如果后续将 `/` 改为敏感接口，再一并纳入保护范围

拦截结果：

- 未登录访问受保护 HTTP 接口：返回 `401`
- 未登录建立 WebSocket：拒绝连接并返回 `401`

### 7.3 认证工具类

建议新增独立认证服务类，例如：

- `AdminAuthService`

职责：

- 初始化默认管理员凭据
- 读取管理员账号与密码哈希
- 校验登录凭据
- 更新账号密码
- 生成与校验密码哈希

这样可以把认证逻辑从 `HttpVerticle` 中拆开，避免该类继续膨胀。

### 7.4 输入校验与错误处理

统一规则：

- 账号不能为空
- 密码不能为空
- 新密码至少 4 位

错误返回：

- 参数错误：`400`
- 未登录：`401`
- 账号或密码错误：`401`
- 服务异常：`500`

错误信息要求：

- 登录失败统一使用模糊错误文案
- 修改凭据失败可返回明确字段校验错误，便于前端提示

## 8. 前端设计

### 8.1 路由结构

新增页面：

- `/login`

保留现有页面：

- `/`
- `/files`
- `/accounts`

### 8.2 登录页

新增独立登录页组件，包含：

- 账号输入框
- 密码输入框
- 登录按钮
- 登录失败提示

行为：

- 打开登录页时先请求 `/auth/session`
- 若已登录，直接跳回首页
- 提交时调用 `/auth/login`
- 成功后跳转首页

### 8.3 登录态守卫

新增前端认证上下文或认证钩子，例如：

- `AuthProvider`
- `useAuth`

职责：

- 获取并缓存 `/auth/session`
- 提供 `login`、`logout`、`refreshSession`
- 暴露 `authenticated`、`isLoading`、`username`

在受保护页面的公共层增加守卫组件，例如：

- `ProtectedAppShell`

行为：

- 加载时先检查登录态
- 未登录则跳转 `/login`
- 登录态未确认前显示加载态

由于当前前端是客户端交互为主，这一版采用前端守卫即可。真正的安全边界由后端 `401` 拦截负责。

### 8.4 设置页中的“账号安全”

在现有设置弹窗中新增页签：

- `Security`

页签中新增表单：

- 新账号
- 新密码
- 确认新密码
- 提交按钮

交互要求：

- 新账号不能为空
- 新密码不能为空
- 两次输入的新密码必须一致
- 提交成功后给出成功提示
- 提交失败时展示后端返回错误

### 8.5 退出登录

在现有头部区域新增退出登录入口，可放在：

- 设置弹窗中的账号安全页签
- 或 Header 的工具区

推荐优先放在 Header 或设置入口附近，保证用户能快速退出。

### 8.6 401 统一处理

前端请求封装已经集中在 `web/src/lib/api.ts`。本次在这里补统一处理逻辑：

- 如果接口返回 `401`
  - 清理本地认证状态
  - 非登录页场景下跳转 `/login`

这样即使某个页面遗漏了前端守卫，也能在调用接口时兜底回到登录页。

## 9. 页面状态设计

### 9.1 登录页状态

- 初始态：可输入账号密码
- 提交中：按钮 loading，避免重复提交
- 失败态：展示错误提示

### 9.2 受保护页面状态

- 会话检查中：显示全局加载态
- 已登录：正常渲染页面
- 未登录：跳转登录页

### 9.3 安全设置状态

- 初始态：展示当前管理员账号
- 提交中：禁用提交按钮
- 成功态：toast 提示修改成功
- 失败态：展示表单错误或服务端错误

## 10. 数据流

### 10.1 首次启动

1. 后端初始化数据库和设置仓库
2. 检查管理员账号与密码哈希是否存在
3. 若不存在，则写入初始账号和密码哈希

### 10.2 登录流程

1. 用户访问受保护页面
2. 前端调用 `/auth/session`
3. 未登录则跳转 `/login`
4. 用户提交账号密码到 `/auth/login`
5. 后端校验成功后写入 Session
6. 前端跳回首页并加载业务数据

### 10.3 修改凭据流程

1. 用户已登录并打开“账号安全”页签
2. 输入新账号、新密码、确认密码
3. 前端提交到 `/auth/credentials`
4. 后端更新设置表中的管理员账号和密码哈希
5. 前端提示成功，当前会话继续可用

## 11. 安全与约束

### 11.1 安全收益

- 未登录用户无法直接使用业务接口
- 未登录用户无法建立业务 WebSocket
- 密码不以明文落库

### 11.2 已知限制

- 初始密码 `admin` 很弱，必须在交付文档中明确建议首次登录后立即修改
- 本次不限制登录失败次数，暴力尝试防护较弱
- 本次不支持忘记密码恢复，只能通过数据库或初始化方式介入
- 只有单管理员账号，不支持审计和操作归属

## 12. 测试方案

### 12.1 后端验证

- 首次启动时自动生成默认管理员凭据
- 正确账号密码可以登录
- 错误账号或密码返回 `401`
- 未登录访问受保护接口返回 `401`
- 未登录连接 `/ws` 被拒绝
- 已登录后修改账号密码成功
- 修改后退出，再用新账号密码重新登录成功

### 12.2 前端验证

- 未登录访问首页会被重定向到 `/login`
- 登录页输入正确账号密码后进入系统
- 登录页输入错误凭据显示错误提示
- 设置页可以成功修改账号和密码
- 修改成功后刷新页面仍保持登录
- 退出登录后再次访问业务页会重新跳转登录页

### 12.3 建议执行

- 前端：`npm run check`
- 后端：`gradle test`
- 关键路径手工冒烟：登录、进入首页、修改账号密码、退出、重新登录

## 13. 部署与回滚影响

### 13.1 部署影响

- 新增可选环境变量：`ADMIN_USERNAME`、`ADMIN_PASSWORD`
- 老数据无需手工迁移表结构
- 首次启动会自动补默认管理员设置

### 13.2 回滚影响

- 若回滚到无登录版本，原有设置表中的管理员凭据字段会保留但不会影响旧版本运行
- 本次不涉及破坏性 Schema 变更，回滚成本低

## 14. 实施拆分

建议按以下顺序实施：

1. 后端补管理员凭据初始化与密码哈希工具
2. 后端新增认证接口与统一认证拦截
3. 前端新增认证上下文、登录页与页面守卫
4. 前端在设置弹窗中新增账号安全页签
5. 前端补退出登录入口和 `401` 统一处理
6. 完成类型检查、后端测试和关键路径冒烟验证

## 15. 验收标准

- 未登录时无法进入站内功能，也无法调用受保护业务接口
- 可以使用 `admin / admin` 完成首次登录
- 登录后可以在页面内修改管理员账号和密码
- 修改成功后退出并使用新账号密码重新登录成功
- 页面与接口在未登录、登录、退出、改密后的状态切换正确
